# ADR 0002: Authentication via Cloudflare Access with Google IdP

Status: **Accepted** (2026-04-30, Phase 0.7 of v2 rebuild)
Supersedes: nothing
Related: [`0001-stack.md`](0001-stack.md), [`0003-state-schema.md`](0003-state-schema.md), [`backend-requirements.md`](../../.claude/docs/ai/dk-edge-v2-frontend/backend-requirements.md)

## Context

The current site shows "Welcome, Max" but `index.html:637` is hardcoded and the Sign Out link at `index.html:638` is `href="#"`. There is no real auth, no session check, no identity. Anyone hitting the URL sees the same UI. This is fine when the site is single-user and obscure, but it blocks two things the v2 explicitly needs:

1. **Cross-device state.** To sync placements across iPhone Safari and Mac Chrome, the worker has to know which user it is talking to. Without an identity claim it cannot key into the KV namespace defined in [`0003-state-schema.md`](0003-state-schema.md).
2. **Future multi-user (potentially).** If Max ever lets someone else use the app (a friend, a syndicate co-bettor), each user needs their own state silo. Same authentication primitive should serve both cases.

This ADR pins how authentication works.

## Decision

### Cloudflare Access with Google as the IdP

- **Cloudflare Access** sits in front of the Worker. It is configured per application in the Cloudflare dashboard, not in code.
- **Google Workspace / Gmail** is the identity provider. The Access policy authorizes specific email addresses; today only `max.sheahan@icloud.com` is on the allow list. Adding other emails later is a dashboard change, no code deploy.
- **Worker reads the email from a request header.** Cloudflare Access verifies the user, then forwards the request with `cf-access-authenticated-user-email` populated. Every `/api/*` route requires this header; absent or empty means 401.
- **Sign-out** is a redirect to `/cdn-cgi/access/logout` (Cloudflare's logout endpoint). The frontend has no token to clear.
- **No client-side SDK.** No Firebase Auth, no Auth0 SDK, no NextAuth. The Worker reads a header. That is the entire auth integration on our code's side.

### Worker middleware

```ts
// worker/middleware/auth.ts
export const requireAuth: MiddlewareHandler = async (c, next) => {
  const email = c.req.header('cf-access-authenticated-user-email')?.toLowerCase()
  if (!email) return c.text('Unauthorized', 401)
  c.set('email', email)
  await next()
}
```

Routes mount it: `app.use('/api/*', requireAuth)`. The email is then available to handlers via `c.get('email')`.

### Email canonicalization

Always lowercase before keying into KV. Cloudflare Access returns whatever case Google sent; the canonicalization rule lives in the auth middleware so every consumer gets the same shape. See [`0003-state-schema.md`](0003-state-schema.md) for downstream usage.

### Local dev

Cloudflare Access is enforced at the edge, not by `wrangler dev`. Two paths for local development:

1. **Mock the header** in development. Add a dev-only middleware that injects `cf-access-authenticated-user-email: max.sheahan@icloud.com` when running under `wrangler dev`. Gated by a `c.env.DEV === '1'` check so it never ships to production.
2. **Use a `*-dev` subdomain** without an Access policy. Less common; only used if we want to test against real Cloudflare KV in the worker without local mocking.

We start with path 1.

## Consequences

### What we gain
- **Zero client-side auth code.** No SDK, no token plumbing, no refresh logic. The frontend treats the user as always-signed-in (because Access ensures it before the request reaches the Worker).
- **Real identity.** `cf-access-authenticated-user-email` is verified by Cloudflare. The Worker trusts the header because traffic that does not pass through Access cannot reach the Worker (assuming the Access policy covers the worker route, which is the configuration we will set).
- **Easy multi-user later.** Add an email to the Access policy in the dashboard. KV keys are already partitioned by email per [`0003-state-schema.md`](0003-state-schema.md). No code change.
- **Fast sign-in.** Google OAuth via Access is one-tap on devices already signed into Google. Most flows complete in seconds.
- **Free.** Cloudflare Access is included in the Workers paid plan tier we already have, up to 50 users, which is well above what we need.

### What we accept
- **Vendor lock to Cloudflare.** If we ever leave Cloudflare Workers, we lose the auth gateway and have to add a real auth stack. Acceptable tradeoff: we are already locked in for the worker runtime, KV, and assets binding; auth being on the same vendor is consistent.
- **Header-trust assumption.** The Worker trusts `cf-access-authenticated-user-email` because Access traffic is the only path to the Worker. This is true as long as the Access policy is configured to cover the entire Worker domain (or at minimum the `/api/*` routes). If a misconfiguration ever opens a non-Access path, header spoofing becomes possible. Mitigation: also verify the JWT signature in `cf-access-jwt-assertion` for sensitive endpoints, OR confirm the Access app covers the right path on every deploy.
- **Sign-out is a redirect, not an in-app action.** Clicking Sign Out leaves the SPA briefly. Users see Cloudflare's logout page for a moment then return to the login screen. Acceptable; matches how every Access-protected app works.
- **No anonymous mode.** The Picks page is gated behind sign-in even though `data.json` is public. We accept this for the v2 because cross-device sync is the whole point. If we ever want a public read-only view, we add a separate path that bypasses Access.

## Alternatives considered

### Firebase Auth
- Pros: full-featured, well-documented, multi-IdP.
- Cons: client-side SDK to bundle, server-side token verification to wire up, another vendor to depend on. Was tried in March 2026 in `dk-edge-finder-app/` and abandoned. The April 10 mockups folder still references it but the code path is dead.
- Decision: not chosen.

### Auth0 / Clerk
- Pros: turn-key auth.
- Cons: paid services, third-party SDKs, more weight than necessary for a single-user app.
- Decision: not chosen.

### Custom OAuth with Google
- Pros: full control.
- Cons: token storage, refresh, CSRF protection, cookie management. Real engineering for a problem Cloudflare Access solves with a checkbox. Skill-folder notes from Max's prior work suggest he has tried hand-rolling auth before and regretted it.
- Decision: not chosen.

### iCloud Hide My Email or Sign in with Apple
- Pros: matches Max's primary email (`@icloud.com`) and devices.
- Cons: Cloudflare Access does not ship a built-in Apple IdP; would require custom OIDC config and ongoing maintenance. Google IdP works regardless of the email's domain.
- Decision: not chosen.

### Stay anonymous (current behavior)
- Pros: zero auth code.
- Cons: blocks cross-device state, the explicit v2 goal. Already analyzed in the plan file under the "API entry per browser" section.
- Decision: explicitly rejected.

## Implementation notes (for Phase 1)

- **Configure Access:** in the Cloudflare dashboard, create a self-hosted application at `dk-edge-finder.max-sheahan.workers.dev/*`. Add Google as an identity provider. Add a policy: include emails `max.sheahan@icloud.com`. Save.
- **Verify locally:** before pushing v2, hit the live URL in a private window. Confirm Cloudflare's Google sign-in flow appears, completes, and lands on the worker.
- **Worker middleware:** ship `worker/middleware/auth.ts` with the snippet above. Mount via `app.use('/api/*', requireAuth)` in `worker/index.ts`.
- **Frontend:** the SPA does not implement sign-in UI. It renders the signed-in state directly because Access guarantees the request reached us. The Account tab shows email + avatar from `GET /api/me`. Sign Out is a link to `/cdn-cgi/access/logout`.
- **`/api/me`:** returns `{ email, picture_url }`. Email comes from the header. Picture URL: optional, parsed from the JWT (`cf-access-jwt-assertion`) if Google's `picture` claim is present, otherwise null. The frontend handles a null picture with initials.
- **Dev mock:** see "Local dev" above. Gate the mock on `c.env.DEV === '1'` so production never sees it.
